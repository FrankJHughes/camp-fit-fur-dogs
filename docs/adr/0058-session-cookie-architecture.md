# ADR‑0058 — Session Cookie Architecture

## Status  
Accepted

## Context  
CampFitFurDogs uses a server‑managed session model.  
Authentication produces:

- A **plaintext session token** (opaque, random, 256‑bit)  
- A **SHA‑256 hash** stored in the database  
- A **Session record** associated with an Owner  
- A **session cookie** containing the plaintext token  

Historically, cookie issuance and cookie value computation were scattered across:

- Authentication callback steps  
- Endpoint code  
- Infrastructure helpers  
- Ad‑hoc cookie builders  

This created several architectural issues:

### 1. Cookie value computation was not centralized  
Different parts of the system computed cookie values inconsistently.

### 2. Cookie issuance was mixed with business logic  
Application logic and HttpContext logic were interwoven.

### 3. Cookie flags were not consistently applied  
Security flags (HttpOnly, Secure, SameSite, Path, Max‑Age) were duplicated and sometimes incorrect.

### 4. Cookie opacity was not enforced  
Some implementations risked leaking structured data or using JWT‑like formats.

### 5. Testing was difficult  
Cookie issuance required full callback flows and HttpContext mocks.

The authentication refactor introduced:

- **ImmutableContextBuilder**  
- **Frank pipeline → Application pipeline → Api endpoint**  
- **SessionTokenService**  
- **SessionCookie value object**  
- **Strict layering boundaries**  

A formal architectural decision is required to define how session cookies are computed, issued, and validated.

---

## Decision  
CampFitFurDogs adopts a **deterministic, immutable, boundary‑controlled session cookie architecture**.

Session cookies are:

- Computed in the **Application pipeline**  
- Issued in the **Api endpoint**  
- Validated in **Infrastructure**  
- Governed by strict security rules  

---

# 1. Cookie Value

The cookie value is the **plaintext session token**, generated by:

```
ISessionTokenService.GenerateToken()
```

### Requirements

- 256‑bit cryptographically secure random value  
- Opaque  
- No user data  
- No dots (`.`)  
- Not a JWT  
- Not reversible  
- Not structured  

The Application pipeline returns:

```
CookieValue : string
```

The Api endpoint issues it.

---

# 2. Cookie Name

The cookie name is:

```
cfd.session
```

### Requirements

- Lowercase  
- Stable  
- Not environment‑dependent  
- Not slice‑dependent  
- Not configurable by consumers  

---

# 3. Cookie Flags

The Api endpoint must apply the following flags:

| Flag | Value | Reason |
|------|--------|--------|
| HttpOnly | true | Prevent JS access |
| Secure | true in preview/prod; false in local | Prevent MITM |
| SameSite | Lax | Prevent CSRF while allowing OIDC redirect |
| Path | / | Session applies to entire site |
| Max‑Age | 24 hours | Matches session expiration |

### Rules

- Flags must be applied **only** at the Api boundary  
- Flags must not be computed in Application  
- Flags must not be computed in Frank  
- Flags must not be configurable per request  

---

# 4. Cookie Issuance (Api Layer Only)

The Api callback endpoint is the **only** place where cookies are issued.

### Responsibilities

- Read `CookieValue` from Application pipeline result  
- Issue cookie with correct flags  
- Redirect user  

### Rules

- No protocol logic  
- No business logic  
- No session creation  
- No token hashing  
- No redirect computation  

---

# 5. Cookie Validation (Infrastructure Layer)

Infrastructure validates cookies by:

1. Extracting the plaintext token  
2. Hashing it using `ISessionTokenService.HashToken`  
3. Looking up the Session record by TokenHash  

### Rules

- No plaintext token may be persisted  
- No plaintext token may be logged  
- No partial matching  
- No token decoding (it is opaque)  

---

# 6. Cookie Lifetime

Session cookies use **Max‑Age**, not `Expires`.

### Requirements

- Max‑Age = 24 hours  
- Must match Session expiration  
- Must not drift from Session record expiration  
- Must not be environment‑dependent  

---

# 7. Layering Rules

### Frank Layer  
❌ Must not issue cookies  
❌ Must not compute cookie values  
✔ May normalize claims  

### Application Layer  
✔ Computes cookie value  
❌ Must not issue cookies  
❌ Must not access HttpContext  

### Api Layer  
✔ Issues cookies  
✔ Applies flags  
❌ Must not compute cookie values  

### Infrastructure Layer  
✔ Validates cookies  
✔ Hashes tokens  
❌ Must not issue cookies  

### Domain Layer  
❌ Contains no cookie logic  

---

# 8. Testing Requirements

### Unit Tests  
- CookieValue is opaque  
- CookieValue contains no dots  
- CookieValue contains no user data  

### Pipeline Tests  
- Application pipeline returns correct CookieValue  
- CookieValue matches TokenHash  

### Integration Tests  
- Cookie is issued  
- Cookie flags are correct  
- Cookie is HttpOnly  
- Cookie is Secure in preview/prod  
- Cookie is SameSite=Lax  
- Cookie is opaque  

### Guardrail Tests  
- No protocol logic in Application  
- No business logic in Frank  
- No cookie issuance outside Api  
- No structured cookie values  
- No JWTs  

---

## Consequences

### Positive  
- Deterministic, secure cookie architecture  
- Strong layering boundaries  
- Immutable, testable cookie value computation  
- Perfect alignment with Session Token Architecture  
- Perfect alignment with Authentication Callback Architecture  
- No sensitive data in cookies  
- No JWT complexity  
- No cross‑layer leakage  

### Neutral  
- Requires developers to understand cookie lifecycle  

### Negative  
- Existing cookie logic required migration (completed)  

---

## Summary  
ADR‑0058 defines the **Session Cookie Architecture** for CampFitFurDogs:

- CookieValue computed in Application  
- Cookie issued in Api  
- Cookie validated in Infrastructure  
- Opaque 256‑bit token  
- Strict security flags  
- Immutable, deterministic, builder‑based flow  

This architecture is now canonical for all session cookie behavior.
