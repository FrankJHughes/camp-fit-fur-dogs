# ADR‑0040 — Middleware Ordering for Security and CORS

## Status  
Accepted

## Context  
As the API evolved to include hardened CORS rules, secure session cookies, and standardized security headers, several ordering constraints emerged:

- CORS must run **before** authentication so that browsers receive the correct headers even when authentication fails.  
- Authentication must run **before** the dispatcher pipeline so that handlers receive a validated `CurrentUser`.  
- Security headers must run **after** routing to ensure all responses—including errors—receive the correct headers.  
- Some middleware (e.g., exception handling) must wrap the entire pipeline to ensure consistent error formatting.  
- PR Preview environments require deterministic ordering to avoid environment‑specific behavior.

Incorrect ordering previously caused:

- Missing CORS headers on authentication failures  
- Incorrect security headers on some error responses  
- Inconsistent preflight behavior  
- Browser‑level failures due to missing or misordered headers  

A formal ordering model was required to ensure consistent behavior across all environments.

## Decision  
The system now uses a **deterministic middleware ordering model** with the following sequence:

1. **Exception Handling**  
   - Wraps the entire pipeline  
   - Ensures all errors produce consistent responses  

2. **CORS**  
   - Executes before authentication  
   - Ensures preflight and failed requests receive correct CORS headers  

3. **Security Headers** (initial pass)  
   - Applies baseline headers early  
   - Ensures consistent behavior for all responses  

4. **Authentication & Session Validation**  
   - Validates session cookies  
   - Resolves the authenticated Owner identity  
   - Runs before the dispatcher pipeline  

5. **Routing & Endpoint Execution**  
   - Executes API endpoints  
   - Invokes the dispatcher pipeline for commands and queries  

6. **Security Headers** (final pass)  
   - Ensures all responses—including those generated by downstream middleware—receive the correct headers  

This ordering ensures that:

- CORS headers are always present  
- Authentication failures do not block CORS  
- Security headers are applied consistently  
- Dispatcher pipeline receives validated identity information  

## Consequences  

### Positive  
- Predictable CORS behavior across all environments.  
- Authentication failures return correct CORS and security headers.  
- Security headers are consistently applied to all responses.  
- Dispatcher pipeline receives validated identity information.  
- Reduced browser‑level inconsistencies and preflight failures.  

### Negative  
- Middleware ordering becomes more rigid and must be maintained carefully.  
- Future middleware additions must be evaluated for ordering impact.  
- Some middleware cannot be easily composed without considering upstream/downstream effects.
