# ADR‑0056 — Session Token Architecture

## Status  
Accepted

## Context  
CampFitFurDogs uses server‑managed authentication sessions.  
A session is represented by:

- A **plaintext session token** (sent to the browser)  
- A **SHA‑256 hash of the token** (stored in the database)  
- A **Session record** associated with an Owner  
- A **session cookie** containing the opaque token  

Historically, session logic was scattered across:

- Authentication callback steps  
- Cookie issuance logic  
- Repository implementations  
- Endpoint code  

This created several architectural issues:

- Token generation was inconsistent  
- Hashing rules were duplicated  
- Cookie value computation was not centralized  
- Session creation was mixed with protocol logic  
- Session identity was ambiguous (TokenHash vs SessionId)  
- Testing required full callback flows  
- Guardrails could not reliably enforce opacity or security flags  

The authentication refactor introduced:

- **ImmutableContextBuilder**  
- **Frank pipeline → Application pipeline → Api endpoint**  
- **SessionTokenService**  
- **SessionCookie value object**  
- **SessionId as the aggregate identity**  

A formal architectural decision is required to define how session tokens are generated, hashed, stored, and consumed.

## Decision  
CampFitFurDogs adopts a **deterministic, immutable, builder‑based session token architecture** with the following characteristics:

---

# 1. Token Generation

Session tokens are generated by:

```
ISessionTokenService.GenerateToken()
```

### Requirements

- 256‑bit cryptographically secure random value  
- Encoded as a URL‑safe opaque string  
- Must contain **no user data**  
- Must contain **no dots (`.`)**  
- Must not be a JWT  
- Must not be base64 of structured data  

The plaintext token is **never persisted**.

---

# 2. Token Hashing

The plaintext token is hashed using:

- **SHA‑256**  
- Lowercase hex encoding  
- Exactly 64 characters  

Hashing is performed by:

```
ISessionTokenService.HashToken(plaintext)
```

### Requirements

- Hashing must be deterministic  
- Hashing must be pure  
- Hashing must be performed **before** session creation  
- Hash must be stored in the Session record  
- Hash must be used for lookup  

The hash is the **lookup key**, not the identity.

---

# 3. Session Identity

Session identity is represented by:

```
SessionId : Guid
```

### Requirements

- Surrogate key  
- Never derived from token  
- Never derived from TokenHash  
- Stable across the session lifecycle  
- Safe for logs, events, and debugging  

This avoids:

- Leaking sensitive data  
- Tying identity to hashing algorithms  
- Breaking identity on token rotation  

---

# 4. Session Creation (Application Pipeline)

Session creation occurs **only** inside the Application Auth Callback Pipeline:

```
IImmutableContextBuilder<
    ApplicationAuthCallbackRequest,
    ApplicationAuthCallbackContext,
    ApplicationAuthCallbackContextBuilderResult>
```

### Responsibilities

- Create Session aggregate  
- Persist via repository  
- Commit via Unit of Work  
- Produce:
  - `SessionId`
  - `TokenHash`
  - `CookieValue`

### Rules

- No protocol logic  
- No external HTTP calls  
- No cookie issuance  
- No HttpContext access  

---

# 5. Cookie Value Computation

Cookie value is computed by the Application pipeline and returned as:

```
CookieValue : string
```

### Requirements

- Must be the **plaintext token**  
- Must be opaque  
- Must not contain user data  
- Must not contain dots  
- Must not be a JWT  
- Must not be reversible  

Cookie flags are applied by the Api endpoint, not the pipeline.

---

# 6. Cookie Issuance (Api Endpoint)

The Api callback endpoint:

- Reads `CookieValue` from the Application pipeline result  
- Issues the cookie  
- Applies required flags:
  - HttpOnly  
  - Secure (preview/prod)  
  - SameSite=Lax  
  - Path=/  
  - Max‑Age  

The endpoint must not:

- Generate tokens  
- Hash tokens  
- Create sessions  
- Compute redirect URLs  

---

# 7. Session Lookup

Session lookup uses:

```
TokenHash
```

### Requirements

- Lookup must be exact match  
- Hash must be indexed  
- No plaintext token may be logged or stored  
- No partial matching  

---

# 8. Testing Requirements

### Unit Tests
- Token generation  
- Token hashing  
- Cookie value computation  

### Pipeline Tests
- Session creation  
- TokenHash correctness  
- CookieValue correctness  

### Integration Tests
- Cookie issuance  
- Redirect behavior  
- Session persistence  

### Guardrail Tests
- Cookie opacity  
- Cookie flags  
- No dots in token  
- No user data in token  
- No protocol logic in Application pipeline  
- No business logic in Frank pipeline  

---

## Consequences

### Positive  
- Deterministic, secure session architecture  
- Strong separation of concerns  
- Immutable, testable token lifecycle  
- No sensitive data persisted  
- No JWT complexity  
- Perfect alignment with Authentication Callback Architecture  
- Perfect alignment with Security & Cookie Guardrails  
- Perfect alignment with ImmutableContextBuilder  

### Neutral  
- Requires developers to understand the token lifecycle  
- Requires updates to test harnesses  

### Negative  
- Existing token logic must be migrated (completed)  

---

## Summary  
ADR‑0056 defines the **Session Token Architecture** for CampFitFurDogs:

- Opaque 256‑bit tokens  
- SHA‑256 lowercase hex hashes  
- SessionId as the aggregate identity  
- Immutable builder‑based session creation  
- Cookie issuance at the Api boundary  
- Strict separation of protocol vs business logic  
- Strong security and testability guarantees  

This architecture is now canonical for all session‑related behavior.
